IMPORTANT PRODUCT DECISION

Use a 5-minute burst window.

The product rule is:
Multiple movements of the same task within a 5-minute window are treated as one schedule-editing burst, not as repeated postponements.

Therefore:
BURST_WINDOW_MS = 5 * 60_000
INITIAL_PLACEMENT_WINDOW_MS = 5 * 60_000

Do not use 2 minutes as the default in this implementation.

ARCHITECTURAL DISTINCTION

Do NOT treat normal reschedule burst members as duplicate_ignored.

duplicate_ignored means:
- technical duplicate
- retry
- double-click
- an event that should not have existed

reschedule burst means:
- several real user edits
- user is arranging the schedule
- each raw event is real and must remain preserved

PART 2 — DETECTION LOGIC

For every new task_rescheduled event:

1. Determine initial placement refinement:

Read:
metadata.taskCreatedAt

If it is a valid ISO date:
isInitialPlacementRefinement =
  event.occurredAt - taskCreatedAt <= 5 minutes

If missing or invalid:
isInitialPlacementRefinement = false

Important:
isInitialPlacementRefinement is factual metadata only.
It means:
"the task was moved within 5 minutes of creation, so the user may still be placing it."

It does NOT mean:
- avoidance
- postponement
- planning failure
- discipline issue

2. Detect prior burst events:

Find prior task_rescheduled events with:
- same userId
- same taskId
- eventType = 'task_rescheduled'
- occurredAt >= event.occurredAt - BURST_WINDOW_MS
- occurredAt < event.occurredAt
- id != current event.id
- if event.dateIso is present:
  same dateIso
- if event.dateIso is missing:
  omit dateIso filter and degrade gracefully

Order by occurredAt ascending.

3. Assign burstRole:

If no prior burst events:
- current event metadata.burstRole = 'sole'

If prior burst events exist:
- current event metadata.burstRole = 'final'
- prior events in the same burst should be updated to metadata.burstRole = 'member'

Important:
'final' means "currently latest event in this burst based on known data."
If another movement arrives within 5 minutes later, the previous final becomes member and the new one becomes final.

4. Preserve existing metadata:

Prisma Json update replaces metadata.
So when updating metadata:
- read existing metadata
- spread existing fields
- add:
  burstRole
  isInitialPlacementRefinement
  burstWindowMs: 300000
  burstDetectedAt: new Date().toISOString()
- do not remove:
  duration
  priority
  flexibility
  taskCreatedAt
  any existing fields

For prior events updated to member:
- preserve their existing metadata too.
- only add/update burstRole and burstWindowMs/burstDetectedAt if safe.

5. Fire-and-forget:

The annotation must not block the response.
If annotation fails:
- console.warn with safe metadata only:
  eventId
  taskId
  userId
  dateIso
  error message
- do not throw
- do not change API response
