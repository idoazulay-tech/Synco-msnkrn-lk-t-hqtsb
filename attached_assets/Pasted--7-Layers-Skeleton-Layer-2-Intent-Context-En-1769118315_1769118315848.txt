אני ממשיך את הפרויקט שנבנה לפי הפרומפטים הקודמים:
- 7 Layers Skeleton
- Layer 2: Intent & Context Engine (Pipeline מלא + ContextManager + IntentAnalysis)
- Layer 3: Decision Engine (DecisionOutput + actionPlan + question/reflection)

עכשיו אני רוצה שתבנה Layer 4: Task & Time Engine בצורה מלאה ומעשית,
ובנוסף: להוסיף Front-End מינימלי שמציג ומאפשר להפעיל את כל ה-flow (Layers 1–4) בצורה ויזואלית, פרקטית, ומוכנה להתרחבות.

מטרה מרכזית
כל קלט של משתמש חייב להפוך לאחד מאלה:
- create / edit / delete / reschedule / confirm / learn / log_note
והכל חייב להיות מוצג בפרונט (לא רק console).

דרישות טכנולוגיות
1) שפה: TypeScript
2) Backend: Node + Express
3) Front-End: React + Vite (או אם כבר יש React ב-Replit אז להשתמש במה שקיים)
4) אין DB אמיתי בשלב הזה, רק In-Memory Store יציב (עם אפשרות החלפה בהמשך)
5) Architecture: modular, clean, scalable
6) comments באנגלית בלבד

חובה: Flow מלא מקצה לקצה
הפרונט שולח טקסט לשרת:
POST /api/analyze
body: { "text": "..." }

השרת עושה:
InputLayer -> IntentContextEngine.analyze -> DecisionEngine.decide -> TaskTimeEngine.apply
ומחזיר JSON שמכיל:
- IntentAnalysis
- DecisionOutput
- UpdatedState (tasks, events, schedule timeline)
- UIInstructions (מה להציג, האם לפתוח modal שאלה, האם להציג reflect וכו')

הגדרת Store (חובה)
InMemoryStore חייב להחזיק:
- tasks[]
- events[]
- notes[]
- scheduleBlocks[] (timeline)
- lastQuestion (אם נשאלה שאלה)
- lastReflection
- contextState (מה-ContextManager)

Task Entity (חובה)
Task:
{
  id: string,
  title: string,
  status: "pending|done|canceled",
  mustLock: boolean,
  urgency: "low|medium|high",
  durationMinutes: number,
  scheduled: {
    dateIso: string,
    startTimeIso: string,
    endTimeIso: string
  } | null,
  dependencies: string[],
  createdAtIso: string,
  updatedAtIso: string
}

  id: string,

Layer 4: Task & Time Engine (חובה יכולות)
TaskTimeEngine.apply(decisionOutput, store) מחיל פעולה בפועל על הנתונים.

חובה לתמוך ב-actionType מה-Decision Engine:
1) create_task
2) create_event
3) reschedule
4) cancel
5) inquire
6) log_note

ולתרגם את זה לפעולות CRUD:
- create: הוספה ל-store
- edit/update: עדכון ישות קיימת
- delete/cancel: שינוי status + הסרה מהטיימליין אם צריך
- reschedule: בניית schedule מחדש
- confirm: אם היה question, קליטת תשובה ועדכון ישות/החלטה

Scheduler בסיסי (MVP)
לבנות planner שמייצר לוז יומי פשוט:
- מניח יום עבודה 08:00 עד 22:00
- events מקבלים מקום קבוע לפי זמן
- tasks נכנסים ל-free slots לפי:
  1) mustLock קודם
  2) urgency גבוה קודם
  3) dependencies קודם
- מוסיף buffer של 5 דקות בין בלוקים (configurable)

Dependencies Rules (חובה)
אם task A תלויה ב-task B:
- A לא יכולה להיות לפני B בטיימליין
- אם אין מספיק מקום, להחזיר UIInstructions שמציע:
  - דחייה
  - קיצור
  - ביטול משימה לא must

Reshuffle Planner (חובה)
כשתגיע משימה דחופה באמצע היום:
- להכניס אותה עכשיו
- להפיק 2 פתרונות בלבד:
  Plan A: לקצר בלוקים לא-must (עד גבול מינימום זמן לכל task)
  Plan B: לדחות משימה אחת לא-must לחלון הבא
להחזיר לשכבת UI "planOptions" עם תיאור קצר בעברית.
המשתמש מאשר Plan A או Plan B.
הבחירה נשמרת ב-store כ-decisionLog (להמשך Layer 5).

Conflict Handling (חובה)
אם אין דרך לשבץ:
- אם יש mustLock שלא יכול להיכנס => UIInstructions: stop + שאלה "איזה משימה להזיז"
- אם אין mustLock => UIInstructions: ask עם options
בשום מצב לא להציג למשתמש 10 שאלות.
רק אחת.

השלמת שאלות (Question Modal Flow)
אם DecisionOutput.decision == "ask":
- הפרונט מציג QuestionModal
- המשתמש עונה
- התשובה נשלחת ל:
POST /api/answer
body: { "answer": "...", "questionId": "..."}
השרת משתמש ב-ContextManager + mergeEntities ואז:
Intent -> Decision -> TaskTimeEngine.apply
ומחזיר מצב מעודכן.

UI Requirements (חובה)
הפרונט חייב להציג 5 אזורים:

1) InputPanel
- שדה טקסט + כפתור שליחה

2) AnalysisPanel
- מציג IntentAnalysis בצורה קריאה (כרטיסים)
  inputType, primaryIntent, entities, missingInfo, confidenceScore

3) DecisionPanel
- מציג decision + reason
- אם ask: מציג "waiting for answer"
- אם reflect: מציג ReflectionCard

4) Timeline
- טיימליין יומי: רשימת Blocks לפי זמן

5) TaskList
- רשימת tasks עם:
  - toggle done
  - cancel
  - mustLock indicator

Actions בפרונט (חובה)
כל ישות בטיימליין וב-task list צריכה כפתורים:
- Mark Done
- Cancel
- Edit Duration
- Toggle MustLock
- Reschedule (simple: "move to later today")

הפעולות שולחות:
POST /api/action
{ action: "mark_done", id: "...", entityType: "task" }

ומפעילות Layer 4 ישירות בלי Intent Engine.

זה קריטי כדי שהמערכת תהיה "פרקטית" ולא רק שיחה.
