IMPORTANT PRODUCT ARCHITECTURE UPDATE

Synco must distinguish between four layers:

1. Raw Activity History
Every meaningful user action may be preserved as factual history.
Examples:
- task created
- task completed
- task moved
- time edited
- duration reported
- entity mentioned
- place mentioned
- event mentioned

Raw activity is not the same as a conclusion.

2. Entity Memory
Entities may be stored early when detected because they are anchors for future context.
Examples:
- people: שני, דני, אמא, מאמן
- places: בית, עבודה, חדר כושר
- life domains: עבודה, לימודים, זוגיות, כלב, משפחה
- projects, recurring events, teams, courses, roles

If the user mentions "שני", Synco may store an entity mention:
- entityName: שני
- entityType: person, if clear enough, otherwise unknown_person
- source context: the task/message where she was mentioned

But Synco must NOT infer from one mention:
- emotional relationship
- fear/love/avoidance
- priority rule
- behavioral pattern
- psychological conclusion

Those require accumulated evidence and/or explicit confirmation.

3. Derived Factual Memories
These are factual, bounded memories derived from raw events.
Example:
- task_created → task_intent
- task_completed → task_completion
- task_execution_completed with confirmed duration → actual_duration + planned_vs_actual_duration

Derived memories are allowed when factual and supported by the event.

4. Learned Conclusions / Patterns
These are higher-level claims.
Examples:
- user often underestimates task duration
- user postpones admin tasks
- tasks related to a person receive high priority
- evening tasks often take longer
- user avoids a specific type of task

These must NOT be created from a single event.
They require:
- multiple supporting events
- Memory Integrity Gate
- confidence scoring
- sometimes user confirmation

CORE RULE

Every activity can be stored.
Only reliable evidence can become active memory.
Only accumulated evidence can become a conclusion.

NEW RESCHEDULE POLICY

Task movement is important, but it is noisy.

1. Five-minute edit burst:
Multiple movements of the same task within a 5-minute window should be treated as one scheduling/editing burst, not multiple postponements.

Example:
Task moved:
09:00 → 09:15
09:15 → 09:30
09:30 → 10:00
all within 5 minutes.

Synco should not learn:
"The user postponed the task 3 times."

Synco should understand:
"The user was editing/refining the task placement."

2. Five-minute initial placement refinement:
If a task is moved within 5 minutes after the task was created, Synco should treat this as initial placement refinement.

Meaning:
The user may still be finding the right slot.
This must NOT count as:
- postponement
- avoidance
- planning failure
- discipline issue
- execution problem

At most, later and cautiously, it may support:
"The initial placement may not have been exact."

3. Raw movements may still be preserved.
But reschedule events should not become active learned conclusions or pattern evidence until a burst/collapse policy exists.

4. task_rescheduled should still NOT create normal Qdrant memories yet.
Before adding it to memory derivation, Synco needs a reschedule burst collapse policy.

IMPORTANT DISTINCTION

Do not confuse:
- raw reschedule history
with:
- reschedule pattern learning

For now:
task_rescheduled may remain stored in Postgres as raw activity.
It should not produce active Qdrant learning memories until a collapse/integrity policy exists.
