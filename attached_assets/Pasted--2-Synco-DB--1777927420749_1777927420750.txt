החלטה טכנית:
אם קיימת דרך עדכון יחידנית ב-API, עדיין עדיף ליצור endpoint ייעודי לשמירת שיבוץ כדי למנוע partial save.

צור endpoint חדש:

POST /api/planner/apply-schedule

body:
{
  "userId": "default-user",
  "date": "2026-05-04",
  "items": [
    {
      "taskId": "task-id",
      "startTime": "09:00",
      "endTime": "09:30"
    }
  ]
}

ה-endpoint צריך:
1. לוודא שיש userId.
2. לוודא שיש date.
3. לוודא ש-items הוא array לא ריק.
4. לסנן פריטים חסרים או לא תקינים.
5. להמיר date + startTime/endTime ל-DateTime תקין.
6. לעדכן רק משימות ששייכות לאותו userId.
7. לא לעדכן משימות deleted.
8. לא לעדכן משימות completed אלא אם כבר כך המערכת עובדת היום. עדיף לא לעדכן completed.
9. להשתמש ב-Prisma transaction אם אפשר.
10. להחזיר response אחיד.

response מוצלח:
{
  "ok": true,
  "date": "2026-05-04",
  "userId": "default-user",
  "updatedCount": 3,
  "updatedTasks": [
    {
      "id": "...",
      "title": "...",
      "startTime": "...",
      "endTime": "..."
    }
  ],
  "skipped": [],
  "warnings": []
}

response חלקי או בעייתי:
{
  "ok": false,
  "error": "message",
  "updatedCount": 0,
  "warnings": [...]
}

חשוב:
אם יש שגיאה באמצע, לא להשאיר DB במצב חלקי.
אם transaction לא אפשרי מסיבה טכנית, עצור ודווח במקום לעשות שמירה חלקית.

3. אם הצליח:

   - ודא שה-DayView מציג את הזמנים החדשים
4. אם נכשל:
   - אל תסגור את ה-preview
   - הצג:
     "לא הצלחתי לשמור את הסידור. שום שינוי לא נשמר."
   - אל תשנה את local state כאילו נשמר

חשוב מאוד:
אל תעדכן localStorage כאילו השמירה הצליחה לפני שה-DB החזיר success.
ה-DB הוא מקור האמת בשלב הזה.

אם taskStore עדיין עובד גם עם localStorage:
לא לשבור אותו.
אחרי שמירה מוצלחת, עדכן את ה-store דרך action קיים אם יש.
אם אין action מתאים, בצע refresh מה-API.
אל תיצור state כפול חדש.
