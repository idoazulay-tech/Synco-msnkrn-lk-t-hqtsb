אני ממשיך את פרויקט MA שנבנה לפי הפרומפטים הקודמים:
- Layer 2: Intent & Context Engine
- Layer 3: Decision Engine
- Layer 4: Task & Time Engine + Express API + React UI
- Layer 5: Learning Engine (DecisionLog, Pattern Detection, Rule Proposals, Confidence, Decay) + LearningPanel UI

עכשיו אני רוצה שתבנה Layer 6: Automation Layer בצורה מלאה ומחוברת לכל מה שכבר קיים, כך שכל Action שמוגדר כ External Side Effect יוכל להתבצע בשירותים חיצוניים בצורה בטוחה, יציבה, ושקופה למשתמש.

עקרונות חובה
1) Internal-first: קודם כל MA מעדכן את ה InMemoryStore וה Timeline (Layer 4), ורק אחרי זה מפעיל Automations
2) Idempotency: לא ליצור כפילויות אם המשתמש שלח שוב או הייתה רשת חלשה
3) Job Queue: כל פעולה חיצונית נכנסת לתור Jobs ומבוצעת על ידי Worker
4) Status: כל Job מחזיר סטטוס אמיתי ל UI: queued | running | success | failed | needs_user_action
5) Reliability: retries עם backoff, timeout, rate limiting בסיסי
6) Permissions: מסך Integrations בפרונט עם חיבור וניתוק שירותים
7) Audit Log: כל פעולה חיצונית נשמרת ב auditLog ומוצגת למשתמש
8) Modular: כל Connector הוא מודול נפרד עם interface קבוע כדי להוסיף שירותים נוספים בעתיד בלי לשבור כלום
9) MVP: להתחיל עם Mock Connector יציב, ובנוסף להכין תשתית Google Calendar Connector (scaffold) עם נקודות חיבור, גם אם אין לנו credentials כרגע

טכנולוגיה
- TypeScript
- Backend: Node + Express (קיים)
- Front-End: React (קיים)
- No external DB בשלב הזה, רק In-Memory
- comments באנגלית בלבד

הרחבת Store מרכזי (חובה)
להרחיב את InMemoryStore (Layer 4) עם:
- integrations: { googleCalendar: { status: "connected|disconnected|error", accountEmail?: string } }
- automationJobs: Job[]
- auditLog: AuditLogEntry[]
- idempotencyKeys: Record<string, { createdAtIso: string, jobId: string }>
- lastAutomationErrors: string[]

חיבור ל Flow הקיים (חובה)
ב server/services/flowService.ts אחרי TaskTimeEngine.apply:
1) לקרוא ל AutomationLayer.planExternalActions(updatedState, decisionOutput) כדי להפיק ExternalActions
2) לכל ExternalAction להכניס Job לתור (enqueue)
3) להחזיר ל UIInstructions מידע:
   - jobsCreatedCount
   - lastJobStatusSummary

Action Router (חובה)
router/actionRouter.ts מקבל החלטות:
- אם provider=mock => להשתמש MockConnector
- אם provider=google_calendar => להשתמש GoogleCalendarConnector
externalActionMapper.ts יגדיר מיפוי:
- create_event פנימי -> ExternalAction create event ב google_calendar (רק אם integration connected)
- update_event -> ExternalAction update
- cancel event -> ExternalAction delete
אם integration לא מחובר:
- לא ליצור Job, אלא ליצור AuditLogEntry מסוג needs_user_action עם summary בעברית שמבקש להתחבר
- ולהחזיר UIInstructions שיפתח IntegrationsPanel

Idempotency (חובה)
idempotencyKey נוצר לפי:
provider + operation + entityType + entityId + hash(payload)
אם key כבר קיים ב store.idempotencyKeys:
- לא ליצור Job חדש
- להחזיר jobId קיים

Job Queue + Worker (חובה)
jobQueue.ts:
- enqueue(job): מוסיף ל store.automationJobs עם status queued
- getNextJob(): מחזיר queued לפי סדר
- updateJobStatus()

worker.ts:
- poll loop כל 1 שניה (configurable)
- לוקח job queued -> running
- מפעיל connector.execute(operation, payload)
- אם הצלחה -> success + audit log success
- אם כשל -> retry לפי retryPolicy עד maxAttempts, ואז failed + audit log failed
- אם חסר הרשאות -> needs_user_action

Retry Policy (חובה)
retryPolicy.ts:
- backoff: 1s, 3s, 7s (דוגמה)
- maxAttempts ברירת מחדל 3
- classify errors:
  - transient (retry)
  - permanent (fail)
  - auth (needs_user_action)

Timeouts (חובה)
timeouts.ts:
- לכל provider + operation להגדיר timeout ms

Rate Limiter (MVP)
rateLimiter.ts:
- להגביל ל X פעולות בדקה לכל provider
- אם עברנו, להשאיר job queued לעתיד הקרוב

Connectors (חובה)
Connector interface ב connectors/Connector.ts:
- name
- validateIntegration(store): boolean
- execute(operation, payload): Promise<{ ok: boolean, status: "success|failed|needs_user_action", data?: any, error?: string }>

MockConnector (חובה MVP)
MockConnector צריך "לזייף" הצלחה אמיתית:
- create: מחזיר externalId
- update: מחזיר ok
- delete: מחזיר ok
ואפשר לדמות כשל כש payload כולל מילה "fail" כדי לבדוק retries

Unit Tests (חובה)

5) permanent error -> failed
6) disconnected integration -> needs_user_action + audit log

8) timeout מסווג כ transient ומנסה שוב
